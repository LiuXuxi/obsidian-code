## 项目背景

我们学校的网站群的业务后端（包括主站、招生网、教务系统、图书馆系统等）最初部署在一个由4-6台高性能物理服务器组成的集群上，前期使用Nginx，适用轮询的算法做流量分发，实现一种简单的负载均衡

学校网站部署了多台服务器，但由于有些时段承载大量的访问请求（**选课系统开放的时间、查成绩的时间、招生报名的时段**），数据获取的任务，这些高并发时段，简单的nginx架构就会有问题（具体问题是1.），

不明原因的就会网站崩溃，并且崩溃原因排查起来也比较费时。后来与学校信息中心合作，去维护，完成的这个开发任务。

通过监控各工作服务器的资源使用情况（如 CPU、内存、磁盘 I/O 和网络性能），性能评分，智能调度用户任务到最优服务器，实现动态的负载均衡，降低负载服务器的压力。还可以通过在网站的高并发访问时期，对性能指标的变化率进行分析，提前预测潜在故障，及时采取措施，防止系统崩溃。一旦发生崩溃时可以数据库查询历史趋势，快速定位问题根源。

我主要负责整个系统的数据采集、分布式通信、数据处理以及存储功能，（架构逻辑），

有师弟负责智能调度、动态负载均衡的实现，基于性能评分对所有可用服务器进行排序，选择评分最高且资源满足任务需求的服务器。

项目实施后，网站几乎不崩溃，学生的体验感提升，问题排查效率提高

Q:轮询策略的工作机制

A：Nginx 按配置文件中的 `upstream`列表顺序，依次将请求分配给后端服务器。多有的服务器都是均等的，不会考虑服务器实际负载或性能差异。

Q：轮询策略在负载均衡上的局限性

A：没办法感知到服务器的实时状态，识别不到服务器的高负载状态。比如某台服务器因突发流量导致CPU满载或磁盘IO瓶颈导致响应变慢（比如教务系统选课时段），Nginx仍会继续向其分配请求。导致用户请求被转发到“已过载”的服务器，导致延迟飙升甚至超时失败，尽管集群中可能有空闲服务器。

Q:我们项目的改进

A:通过实时采集各服务器的性能指标（CPU、内存、磁盘IO、网络等），计算综合健康评分。实现将请求动态路由到当前整体性能最好的服务器，避免盲目的轮询转发。

## 项目的基本功能

### 使用到的中间件

使用gRPC+protobuf实现分布式通信

#### gRPC

- 谷歌开源的高性能的RPC框架，基于 HTTP2 协议设计和ProtoBuf序列化协议开发，使用时只需要在本地生成grpc的客户端和服务器端的代码，就可以进行远程调用和远程传输数据。客户端和服务端只需要在此基础上去实现业务层面的内容，不需要去关注底层的通信细节。
- 关键特点

- 支持多语言
- 通信协议基于标准的 HTTP/2 设计，支持双向流、二进制框架和消息头压缩、单 TCP 的多路复用特性，这些特性使得 gRPC 在发送和接收方面都非常紧凑和高效
- 序列化默认使用protobuf，Protobuf是一种高效的二进制消息格式，序列化的速度很快，基于 HTTP/2 + PB,实现高性能。
- 超时取消机制。gRPC允许客户端指定他们愿意等待多长时间完成一个RPC。服务器可以决定在超时后取消。防止资源浪费。

- 实现步骤

- IDL文件定义要传输的数据的结构和grpc服务，消息包含一系列的名称-值对，定义的gRPC服务指定可以远程调用的方法，参数和返回类型。
- 编译，proto消息使用自带的protoc工具，服务端骨架和客户端存根还需要额外的gRPC插件

- 服务端骨架提供服务端方法的默认实现框架，客户端存根提供远程方法的本地调用接口

- 在服务器端，服务器实现这个接口并运行一个gRPC服务器来处理客户端调用。在客户端有一个存根（封装了序列化反序列化、HTTP/2等通信细节），提供与服务器相同的方法，
- 客户端可以在本地直接调用这些方法，并且这些方法将调用的参数包装在适当的protobuf消息类型中，将请求发送到服务器。服务器收到请求消息后，服务器会通过protobuf解译proto文件，还原出最先定义好格式的数据结构，进行业务处理；数据处理完后，服务器需要使用Protocol Buffers重编译应答数据，通过gRPC协议发送应答消息；
- protobuf消息代码 生成对应字段的C++ API 来读写消息 API包括了set设置标量字段 mutable获取单个嵌套的消息，返回该消息的 可修改指针，直接修改原数据。 add在repeated字段末尾新增一个元素，返回其指针供赋值 （）获取字段值 size获取元素数量 clear清空字段
- 我们只需要实现业务逻辑，调用生成的接口

- protobuf在gRPC框架中的作用

- 定义数据结构
- 定义服务接口
- 通过序列化和反序列化，提升传输效率

##### gRPC的优势

- 项目需要实时采集系统资源信息，并通过 RPC 发送到服务端。gRPC默认 使用 Protobuf 进行序列化，因为是二进制的数据，更紧凑、传输更快。HTTP/2 协议提供了多路复用、头部压缩等特性，提升通信效率。Protobuf + HTTP/2 = 高性能通信
- 本项目采用 C++ 实现后端服务和客户端。gRPC 对 C++ 支持完善，提供了丰富的 API 和示例代码，开发体验优秀。还支持跨平台
- 由谷歌维护，社区活跃度高。使用人数很多，稳定性安全性兼容性都比较好，实际开发问题没那么多

###### HTTP/2协议

- 支持长连接和双向流式传输。流式传输允许 客户端和服务端在单个连接上发送或接收多个消息序列，而不是像传统 RPC 那样一个请求对应一个响应。天然支持四种流式模式，适用于实时场景

- HTTP/1.x 难以支持长连接和流式数据传输。

- 多路复用：通过流式传输，允许在单个 TCP 连接上同时发送多个请求和响应。实现高并发RPC，客户端可以同时发送多个 RPC 请求，无需等待前一个完成，服务器可以并行处理多个 RPC 调用，提高吞吐量，减少延迟

- 避免了HTTP/1.x队头阻塞问题，对头阻塞问题会致同一连接无法并行处理多个请求，必须串行。

- 二进制协议：使用二进制格式的**帧（Frame）**（如 `HEADERS`、`DATA` 帧）传输，紧凑高效。二进制分帧机制维护了一个消息边界，接收方可以正确地分离每个帧，解决TCP粘包问题；二进制格式传输，数据量更小，降低网络带宽消耗，RPC调用速度更快。
- 这里结合ProtoBuf 二进制序列化，进一步减少传输体积。实现高效编解码

- HTTP/1.x 基于文本（如JSON），解析效率低且冗余（多余字符、换行等）。

- 采用HPACK算法头部：通过静态/动态表缓存头部字段，减少冗余头部传输的额外开销，提高传输速率。

- HTTP/1.x 的头部（Header）重复发送（如 `Content-Type`、`Authorization`），浪费带宽。

- 调试工具较少，调试复杂；生成的代码多，增加了编译时间和二进制大小，而且官方文档很少，社区活跃度低，很多特性没有说明

- 主要面向腾讯生态，偏向企业级服务治理，整体架构复杂，不适合小型项目或非腾讯技术栈项目

- BRPC(百度)

- 百度开源，主要面向 C++，文档中文居多，国际社区影响力弱，不适合跨语言协作场景

- Dubbo（阿里）

- 专为java语言开发，其他语言的功能不完整，社区支持有限，性能上也不如gRPC（高并发场景下的延迟和吞吐量表现）

- srpc(搜狗)

- 相对较新，缺乏生产环境验证，稳定性的风险比较高，社区和生态系统不够完善

###### 项目中的使用

- .proto定义了gRPC服务的两个调用方法 还有消息的数据结构

- 使用`service` 关键字定义服务
- 每个RPC方法使用 `rpc` 关键字声明
- 方法名，包含参数和返回值

编译生成文件有（客户端存根Stub和服务端骨架Service）。包含服务接口定义。还有proto消息类。

- 服务端骨架（Service）只提供基础实现的框架，需要实现这两个RPC服务接口方法的具体逻辑

- 用于接收工作服务器（监控采集端）上报的最新监控数据的方法。客户端调用该方法时，服务端会清空当前存储的数据并用请求中的数据更新 ，并使用map映射结构存储不同服务器的数据加以区分，通过互斥锁保证线程安全的并发读写，最后返回OK状态表示处理成功。protobuf消息类型封装了最新一次收到的全部系统监控的指标
- 用于管理服务器获取gRPC服务器端当前保存的最新监控数据的方法。客户端调用该方法时，服务器会将当前保存的 监控数据拷贝到响应中，返回给客户端。这样客户端就能获取到最新的监控数据。
- gRPC服务器启动

- 设置要监听的地址和端口
- 创建 `grpc::ServerBuilder`gRPC提供的配置类，配置服务器参数，这里还会指定连接的安全方式，比如项目目前使用grpc::InsecureServerCredentials()的非加密的连接。如果考虑安全性可以换成带 TLS 的证书认证
- 再把我们写好的具体业务逻辑的服务接口注册进到这个 ServerBuilder 上。这一步就相当于对外暴露接口，供客户端使用
- 调用 `BuildAndStart()` 方法启动服务器，还要wait阻塞主线程等待请求,保持服务器持续运行。

- 客户端存根，主要提供本地调用方法的接口，封装了底层网络通信细节（序列化反序列化、网络传输、错误处理、超时控制），客户端直接调用其方法。

- 首先要和 gRPC 服务器建立通信通道（channel），指定了服务器地址和认证方式。这个通道负责底层的网络连接和数据传输。有了channel之后，客户端需要构建生成的stub存根对象，里面封装了所有可用的远程方法接口。每次通过stub发起远程调用的时候，可以设置超时时间等来管理本次调用的 生命周期
- 客户端调用发送数据的方法是，将本地构造的protobuf消息类型 数据发送到服务器，实现监控数据的上报。调用后会检查返回状态，判断本次调用是否成功。
- 客户端调用获取数据的方法是，从维护的多个RPC通信通道，分别连接不同的工作服务器，多线程并发拉取多台服务器数据，统一上传至管理服务器。调用后检查返回状态，判断本次调用是否成功
- 后续通过 stub 调用服务方法，执行 RPC 调用，实现数据的远程发送和接收

#### protobuf

- protobuf是 Google 的跨语言跨平台可扩展的序列化和结构化数据的工具，有了远程通信的框架，确定了要传输的消息的数据结构后，就要序列化与反序列化。
- 序列化就是将数据结构或对象转换成二进制串的过程。转换为二进制串后才好进行网络传输；反序列化是将在序列化过程中所生成的二进制串转换成数据结构或者对象的过程。将二进制转换为对象才好进行后续处理
- OSI七层协议模型中展示层 的主要功能就是是把应用层的对象（class类等）转换成一段连续的二进制流，然后再把这个二进制流通过TCP/IP协议在网络中进行传输，或者反过来，把接收到的二进制流转换成应用层的对象，

- TCP/IP协议的应用层对应OSI七层协议模型的应用层，展示层和会话层，所以序列化协议属于TCP/IP协议应用层的一部分。
- 二进制串：序列化所生成的二进制串指的是存储在内存中的一块数据。例如，C++语言的字符串可以直接被传输层使用，因为其本质上就是以’\0’结尾的存储在内存中的二进制串

- 序列化反序列化的组件

- IDL文件（.proto）：与语言和平台无关的接口描述语言，相当于通信双方的一个契约，定义分布式系统中要传输的消息的数据结构和相关接口
- IDL Compiler(protoc编译工具)：IDL文件中约定的 内容为了在各语言和平台可见，需要有一个编译器，将IDL文件转换成各语言对应的动态库，这样子就生成了客户端和服务端可以使用的代码。[将 .proto文件编译为可执行文件 .pb.h .pb.cc .grpc.pb.h .grpc.pb.cc]
- Stub/Skelet on Lib：stub是 在分布式系统客户端的代码，对应用层暴露定义好的序列化的接口，客户端会调用本地stub方法接口，一方面接收应用层的参数，并对其序列化后通过底层协议栈发送到服务端，另一方面接收服务端序列化后的结果数据，反序列化后交给客户端应用层；Skeleton部署在服务端，其功能与Stub相反，从传输层接收序列化参数，反序列化后交给服务端应用层，调用server的实际方法，并将应用层的执行结果序列化后最终可以传送给客户端Stub。内部封装好的，只需要调用接口。
- Client/Server：指的是应用层程序代码，他们面对的是IDL所生成的特定语言的class或struct。
- 底层协议栈和互联网：序列化之后的数据通过底层的传输层、网络层、链路层以及物理层协议转换成数字信号在互联网中传递。

- gRPC 的默认序列化协议

- 整个项目的构成大多使用谷歌的工具链，protobuf和grpc两个核心工具都是谷歌开源的，谷歌的代码风格比较好。
- gRPC 官方推荐使用 Protobuf 作为默认的接口定义语言（IDL）和序列化机制
- 关键优势在于：

- 二进制编码，体积小，速度快，比 JSON 快 3 倍，体积小 50%
- 强类型约束：通过 `.proto` 文件明确定义接口和数据结构，编译时会检查类型合法性，避免运行时出错。
- 可以直接新增字段兼容旧服务，可扩展性强
- PB 提供多语言代码生成工具。

##### **对比XML JSON Protobuf**

传输效率层面

- XML、json序列化之后是一个文本，本身的数据量大，占用更多的带宽，传输时间长。protobuf序列化之后成二进制流，数据量更小，数据描述文件大小只有1/10至1/3，传输过程中带宽消耗小，大容量数据传输中不容易丢包。
- 而且protobuf二进制编码可直接映射到内存结构，解析速度比XML快20-100倍,XML、JSON进行数据编译时，数据文本格式更容易阅读，但进行数据交换时，解析文本需要大量的CPU计算，自然会影响整个传输速率。

- 在本项目中，监控系统需要频繁采集并传输 CPU 负载、内存、网络等数据，使用 Protobuf 可以显著减少带宽占用和处理开销。

支撑跨平台、多语言层面

- 通过统一的IDL（接口描述语言）生成类型安全代码，编译时就可以发现类型错误；相比之下JSON是弱类型，XML结构比较复杂容易出错
- Protocol Buffers可编译性强。自带一个编译器，proto文件通过编译器进行编编译生成一个类似库文件，基于库文件才能真正开发数据应用。Protocol Buffers支持跨语言编译，可生成目标语言的代码，提高实际开发效率（由于现网中负责网络设备和服务器设备的运维人员往往不是同一组人，运维人员可能会习惯使用不同的编程语言进行运维开发，那么Protocol Buffers其中一个优势就能发挥出来——跨语言）
- 协议的扩展性好，因为proto每个字段都有一个 唯一数字编号，而不是直接使用字段名序列化，新增字段不会影响旧版解析，而XML和JSON使用字段名序列化，修改起来复杂

------------------------中间件的总结------------------------------------------

### 监控数据定时采集

- 每个工作服务器都包含一个监控组件，每隔十秒去获取各自系统上的各类最新的性能数据 ，逐个获取并将采集结果封装成一份总的protobuf监控数据结构作为gRPC的消息体，并将此次采集到的完整监控数据通过RPC客户端上报到服务端。

- Linux 内核默认不会暴露所有内部符号（变量、函数），所以用户空间是默认不能访问内核数据的，用户态程序运行在低特权级别（Ring 3），无法直接调用内核空间（Ring 0）的函数。

###### 采集方式

- /proc低频数据，内核维护的数据本身的变化频率低（内存、负载），直接读取/proc文件系统的方式，获取内核导出的统计信息。/proc是内核官方的接口，兼容性好、实现**简单**，直接解析其中的文本就可以。采集延迟的性能也足够。通常50-100ms。更适合小规模数据

- 缺点：通过read()读取/proc文件的方式获取，内核都要把原始二进制数据**格式化为文本**，用户空间还要再**解析**文本为结构体或数值，**增加了 CPU 开销**。特别是内核要把数据内容**拷贝**到用户缓冲区，对于监控采集频率较高的情况，**拷贝开销高。**

- 内核模块mmap,高频数据，内核维护的数据本身的变化频率高（CPU软中断、cpu使用情况、），采集逻辑是通过自定义实现内核模块（直接采集内核内部的详细数据，通过 mmap 机制将内核内存映射到用户空间），根据监控需求定义合适的结构，并在内核分配一块内存，存放采集到内核存放的各系统指标的原始数据，作为内核和用户空间共享的数据区，再通过定时器实现每秒定时获取，接下来要暴露到用户进程，就要将内核模块注册为字符设备，并实现mmap回调，这样用户空间可以通过mmap将内核分配的内存区映射到用户空间，后面首次访问该虚拟地址时，操作系统通过缺页异常，将这段虚拟地址和内核的物理内存建立映射关系（页表项），之后无需再次拷贝，用户就可以直接访问映射区内存实时获取到最最新更新的内核数据，用户空间访问的就是内核分配的真实物理内存的映射。实现用户与内核的高效实时交互。
- 选择内核模块mmap采集的原因

- 项目推进过程中发现，要实现更低的性能开销和低延迟，通常延迟1-5ms，可以使用内核模块mmap方式，
- mmap内存映射避免内核态到用户态的数据拷贝还有文本格式化的过程，零拷贝，开销极低，最大限度降低采集延迟，性能特别高，几乎没有额外的开销，非常适合高频采集、实时大数据量的监控场景。
- 相比eBPF少一层虚拟机开销
- 缺点：需要内核编程，开发难度高点。  
    存在安全风险：内核模块可以按照需要加载和卸载扩展内核的功能，但这是完全的内核编程，存在**安全性**的问题，如果内核代码崩溃，整台机器及其上运行的所有程序都会崩溃。因为内核是特权代码，它可以访问机器上的所有内容，包括所有数据，因此内核中的恶意代码将是一个严重的问题。

- eBPF程序eBPF 程序运行在内核的受限虚拟机中，可以动态挂载到内核的网络协议栈的 hook 点（系统调用等），比如这里TC Hook在协议栈入口抓包，延迟<1ms，在内核态实时采集和处理数据，并通过 eBPF map 与用户空间交互获取数据。eBPF 可以直接挂载到网络协议栈的关键路径（如 TC 或 XDP），对每个经过的数据包进行实时统计和处理

- 能在不更改内核代码的前提下，实时获取和修改操作系统的行为。像插件一样扩展内核功能
- eBPF 程序可以在系统运行时随时加载和卸载，不需要重启内核或影响现有业务。。
- 安全性好，eBPF 运行在受限的沙箱环境，而且eBPF 验证器，确保只有在安全运行的情况下才能加载 eBPF 程序——它不会导致机器崩溃或陷入死循环，也不会允许数据被泄露。
- 适合网络流量、包级别的实时监控，能做到极低的性能开销和高频低延迟采集。
- bcc、libbpf、bpftrace 等工具链完善，开发和集成门槛低，社区有大量成熟案例可借鉴。
- 缺点：BPF 程序虽然运行在内核态，但是在一个受限的虚拟机中执行，所有指令都要经过验证和解释，相比直接用内核模块访问内核数据结构，存在一定的虚拟机开销，**性能**略低于内核模块mmap

###### **TC和 XDP**

- 都是 Linux 内核里用来处理网络数据包的技术
- TC（Traffic Control）

- TC 是 Linux 网络协议栈中的一个流量控制框架，主要用来做流量整形、限速、包过滤等。
- TC 的 hook 点位于网络协议栈和网卡驱动之间的“中间层”。也就是数据包已经经过内核协议处理，但还没在出方向真正发到网卡或者在入方向刚从网卡收到。

- 入方向（ingress）：数据包从网卡收到后，先经过 TC ingress，再进入内核协议栈（如 IP 层、TCP/UDP 层等）。
- 出方向（egress）：数据包在内核协议栈里被应用或系统生成后，先经过 TC egress，再发到网卡驱动，最终通过物理网卡发送出去。

- 用 TC 挂载 eBPF 程序，可以对进出网卡的数据包做各种统计、过滤和处理，适合需要更多协议信息和上下文的场景。
- 网络流量统计可以用 TC

- XDP（eXpress Data Path）

- XDP 是 Linux 内核里专门为高性能网络处理设计的一个框架。
- 它的 hook 点在网卡驱动的最前面，数据包刚到网卡、还没进入内核协议栈时就能被处理。
- 用 XDP 挂载 eBPF 程序，可以极快地处理、丢弃、转发数据包，延迟极低，非常适合高性能、低延迟的网络监控和防护场景。
- 需要极致的性能可以用 XDP

- 系统调用的方式，比如通过 `sysinfo`、`getrusage`、`ioctl` 等系统调用来获取系统资源信息，这种方式确实可行，很多基础的监控工具也会用到。

- 但系统调用的方式通常功能有限，有些只能获取到部分的全局资源信息，统计数据不全面，比如整体的内存、CPU 使用率等，无法细致到每个 CPU、每块网卡、每类软中断等详细指标。满足不了监控的数据需求，而且本质上也还是基于读取/proc文件或者内核的变量
- `sysinfo`获取系统整体的内存、交换空间、负载等全局信息
- `getrusage`获取当前进程或其子进程的资源使用情况

##### 具体实现

为什么统计计这些指标，对于这个项目来说的意义

###### cpu负载信息的采集 （/proc）

- 从/proc/loadavg中读取一行数据，取出前三个字段作为过去1/5/15分钟的平均负载值，最后将这些值填充到统一的pb消息体的cpu负载字段，后续可通过gRPC传输。
- `/proc/loadavg`存储的信息

- 文件内容是一行文本，前三个是平均负载信息，内核通常每五秒更新一次数据，后两个数据分别是在采样时刻，系统中活跃的任务的个数和最近一个新创建的pid值，包括轻量级进程即线程。前三个数据足以反映当前系统的繁忙情况。

- 统计指标的意义

- 平均负载是指单位时间内均活跃进程数，它包括了系统处于可运行状态和不可中断状态的进程数，它不仅包括了正在使用 CPU 的进程，还包括等待 CPU 和等待 I/O 的进程。
- 可以反映系统整体的负载情况，看负载可以对整体服务器的压力有一个把握，及时发现和预警系统过载问题
- 一旦负载过高，就可能导致进程响应变慢，进而影响服务的正常功能。
- 平均负载最理想的情况是等于 CPU 个数。超过CPU个数系统出现了过载。
- 分析方法

- 综合来看这三个时间段的值，根据历史数据判断负载的变化趋势
- _如果 1 分钟、5 分钟、15 分钟的三个值相差不大的话，就说明系统负载很平稳。_
- _但如果 1 分钟的值远小于 15 分钟的值，就说明系统最近 1 分钟的负载在减少，而过去 15 分钟内却有很大的负载。_说明在这14分钟的时间里系统的负载情况是有一点问题的，_我们可能后续就需要把这个数据读取出来看看这14分钟内有哪些进程导致了我们负载的升高。_
- 反过来，如果 1 分钟的值远大于 15 分钟的值，就说明最近 1 分钟的负载在增加，这种增加有可能只是临时性的，也有可能还会持续增加下去，所以就需要持续观察。
- 一旦 1 分钟的平均负载接近或超过了 CPU 的个数，就意味着系统正在发生过载的问题，这时就得分析调查是哪里导致的问题，并要想办法优化了。

- 平均负载高有可能是 CPU 密集型进程导致的，还有可能是 I/O 更繁忙了
- 平均负载提供了一个快速查看系统整体性能的手段，反映了整体的负载情况。但只看平均负载本身，我们并不能直接发现，到底是哪里出现了瓶颈。这时候需要根据一些其他的系统数据来辅助分析负载的来源。需要再结合其他指标（如 CPU 利用率、内存、IO 等）进一步分析，去判断是由哪些导致它负载过高还是什么其他的情况

###### 选择proc方式采集的原因

- 数据反映的是长期的趋势不是瞬时状态。本身变化频率低，约5秒更新一次。对采集实时性的要求不高，性能完全足够
- 数据很小只有一行，拷贝和解析开销也很小
- 内核提供的标准化接口，实现简单不需要额外的开发，直接读取即可
- 需要更高性能的场景，如Android的LMK(低内存杀手)，也仍然基于/proc数据做出决策，足以证明性能是足够的
- 前期调研

- 已经成为linux生态中公认的性能监控的标准接口，几乎所有的主流Linux性能工具都基于/proc实现

- top/htop：通过解析`/proc/stat`、`/proc/[pid]/stat`等文件获取CPU和进程信息
- free：直接读取`/proc/meminfo`显示内存使用情况
- ps：扫描`/proc/[pid]/`目录下的各种状态文件
- 工具的广泛应用证明了/proc接口的可靠性和实用性

- Android作为Linux衍生系统，也大量使用/proc获取硬件和性能数据
- 基于Android 13 (T)版本的源码

- 通过`/proc/[pid]/smaps`获取进程详细内存信息，计算关键内存指标，用于Android应用内存监控
- 读取`/proc/stat`获取全局CPU使用信息，解析计算各状态(用户态/内核态/空闲等)的时间占比，用于系统负载监控
- 低内存杀手的实现（高性能场景），读取/proc/zoneinfo获取内存区域信息，监测各内存zone的压力，根据`/proc/[pid]/oom_score_adj`计算进程优先级，可以在内存紧张时选择性地终止进程
- 通过`/proc/cpuinfo`获取处理器型号、核心数等硬件数据
- 通过`/proc/meminfo`获取系统内存总量、使用量等
- Android框架层通过`/proc/[pid]/`目录获取进程内存占用信息、线程数等指标
- 还有很多硬件信息也通过/proc获取，比如电池状态、wifi信息等
- 成熟度高

###### cpu软中断信息的采集

- 实现了一个内核模块，使用 __get_free_pages() 在内核空间分配了一块 PAGE 对齐的结构体数组内存，用于存放所有 CPU 的软中断统计数据。定时器回调函数中，使用 for_each_possible_cpu() 遍历所有 CPU，通过 kstat_softirqs_cpu() 内核 API 读取各类软中断计数，填充到分配的数组内存中。高精度定时器（hrtimer）每秒触发一次，实现数据的实时更新。再注册一个 /dev/cpu_softirq_monitor 字符设备节点，获取文件描述符，实现mmap回调，将存储到统计数据的那块物理内存暴露给用户空间。用户空间就打开设备节点获取到文件描述符后，使用mmap将内核中的数据映射到自己的虚拟地址空间。每次采集时，直接 遍历映射区即可获得所有 CPU 的最新软中断统计数据。记录本次采集的时间点；如果之前已经采集过该 CPU 的数据，则计算与上次采集之间的时间差和计数差，然后将差值除以时间间隔，得到单位时间内的软中断发生频率，采集到的数据填充到统一的 protobuf 消息体的CPU软中断字段，后续可通过 gRPC 传输。
- 数据在内核存储位置

- 软中断的数据统计由内核通过 per-CPU 变量维护，可以通过 kstat_softirqs_cpu(softirq_type, cpu) API 访问。NR_CPUS 是内核编译时配置的最大可能 CPU 数量；NR_SOFTIRQS 是软中断类型的数量（目前为 10 种）。每个 CPU 对应一组计数器，记录该类型软中断在该 CPU 上的触发次数。这些计数器由内核在软中断处理时自动更新，包含了各类软中断的累计计数。

我们的内核模块使用 for_each_possible_cpu() 遍历所有可能的 CPU，通过 kstat_softirqs_cpu() API 读取以下 10 种软中断的统计数据：

- `hi`（HI_SOFTIRQ，高优先级软中断），用于处理一些需要尽快完成的高优先级任务。实际应用中很少用到，主要为内核预留。
- `timer`（TIMER_SOFTIRQ，定时器软中断）负责处理内核定时器到期后的回调任务，比如定时事件、周期性任务等。是内核定时机制的基础。
- `net_tx`（NET_TX_SOFTIRQ，网络发送软中断处理），网络数据包的发送任务。比如协议栈准备好数据后，通过软中断机制异步完成实际的数据包发送。
- `net_rx`（NET_RX_SOFTIRQ，网络接收软中断），处理网络数据包的接收任务。当网卡收到数据包后，内核通过软中断机制将数据包从驱动转交给协议栈处理。
- `block`（BLOCK_SOFTIRQ，块设备软中断），处理磁盘等块设备的异步 I/O 操作。比如磁盘读写完成后，通过软中断通知内核进行后续处理。
- `irq_poll`（IRQ_POLL_SOFTIRQ，中断轮询软中断）用于高效处理大量中断的场景，常见于高性能网络设备。通过轮询方式批量处理中断，减少中断开销。
- `tasklet`（TASKLET_SOFTIRQ，tasklet 软中断）内核中一种轻量级的延迟任务机制，适合需要在软中断上下文中延后执行的任务。tasklet 是基于软中断实现的。
- `sched`（SCHED_SOFTIRQ，调度软中断）与内核调度相关的软中断，用于处理调度器的延迟任务，比如负载均衡、进程唤醒等。
- `hrtimer`（HRTIMER_SOFTIRQ，高精度定时器软中断）处理高精度定时器事件，适用于需要亚毫秒级定时精度的场景，比如实时任务调度。
- `rcu`（RCU_SOFTIRQ，RCU 软中断）用于内核 RCU（Read-Copy-Update）机制的回收和同步任务。RCU 是一种高效的读多写少同步机制，广泛用于内核数据结构的并发访问。

- 软中断结构体与注册机制

- 在 Linux 内核中，软中断的处理是通过一个结构体 struct softirq_action 来实现的。这个结构体里面保存的是软中断的处理函数。内核中还有一个全局数组 softirq_vec[NR_SOFTIRQS]，它用来存放所有软中断类型的处理函数。

- 内核数据采集逻辑

- 如何触发：当内核或者驱动需要处理一些不需要立即执行的任务时，它会调用 raise_softirq() 函数来触发一个软中断。比如：当网络数据包到达时，内核会调用 raise_softirq(NET_RX_SOFTIRQ) 来标记这个软中断，也就是设置它的 pending 标志位，相当于一个待处理的标志，内核就知道这个软中断还没处理，等下要处理它。
- 处理时机：软中断的主调度函数是 do_softirq()，当某个软中断被标记为 pending 后，内核会在适当的时候调用这个函数来处理它。具体来说，软中断的处理时机有几种：

- 在硬中断结束后：比如在 irq_exit() 中调用 do_softirq()；
- 在中断上下文中：如果当前正在处理硬中断，那么软中断会被延迟到退出中断上下文后再处理；
- 通过内核线程 ksoftirqd ：如果软中断频繁触发（比如超过 10 次），系统就会唤醒 ksoftirqd 线程来处理它。
- 所以，软中断并不是立刻处理，而是根据系统状态，在合适的时间点去处理。

- 如何统计：在 __do_softirq() 的处理循环中，每当某类软中断即将被执行时，内核会增加该类型软中断的计数器。这样，每当某类软中断被调度执行时，对应的 CPU 上的计数器就会被更新一次，从而实现对软中断的分 CPU、分类型的实时统计。我们的内核模块通过 kstat_softirqs_cpu(softirq_type, cpu) API 来读取这些统计数据。
- 这样，每当某类软中断被调度执行时，对应的 CPU 上的 kstat_softirqs[cpu][softirq_type] 计数器就会被更新一次，从而实现对软中断的 分 CPU、分类型 的实时统计。

- 指标统计的意义

- 概念

- 首先，中断其实就是让 CPU 能够及时响应各种异步事件，比如外设输入、网络包到达、定时器到期等。
- 但中断处理时间不能太长，如果在中断上下文里做太多事，会影响系统对其他事件的响应，导致实时性变差，甚至引发系统卡顿。
- 为了解决这个问题，Linux 把中断处理分成了两部分：  
    上半部分叫硬中断，负责最紧急、最核心的工作，尽快完成；  
    下半部分就是软中断，负责把剩下的、可以延后处理的任务异步处理掉，这样既保证了响应速度，又不会让 CPU 被中断长时间占用。
- 每个 CPU 都可以独立运行软中断，实现并行处理。

- 分析方法

- 当我们发现 CPU 的软中断使用率增高时，说明内核在处理大量的异步任务，这可能会导致像网络收发变慢、系统调度延迟等性能问题。但光知道软中断高还不够，我们还需要进一步分析到底是哪一类软中断出了问题。这个时候，我们就可以通过统计每种类型软中断的触发频率，比如网络收发（net_rx、net_tx）、定时器（timer）、块设备（block）等，来看看到底是哪一类软中断突然激增。这样就能快速定位到是网络、定时器还是磁盘等哪个环节导致了性能瓶颈，从而有针对性地去优化和排查问题
- 正常情况下，同一种软中断在不同 CPU 上的累计次数应该比较均衡。如果发现所有软中断都集中在一个 CPU 上，其他 CPU 很闲，就说明负载分布不均，可能导致单核过载、系统吞吐量下降，这时候就需要优化中断亲和性或者系统调度策略，让负载更加均衡。

###### cpu状态信息的采集

CPU 状态信息的采集分为内核空间和用户空间两部分协同完成：

- 内核模块（cpu_stat_collector.ko）：

- 内核模块通过高精度定时器（hrtimer）每秒遍历所有 CPU，读取 kcpustat_cpu(cpu).cpustat[] 数组中的数据，获取自系统启动以来每个 CPU 在各类状态下的累计时间（user、nice、system、idle、iowait、irq、softirq、steal、guest、guest_nice）
- 对于 idle 和 iowait 时间，优先使用 get_cpu_idle_time_us() 和 get_cpu_iowait_time_us() 获取精确值（考虑 NO_HZ tickless 模式），若不可用则回退到传统方式
- 采集到的原始累计值（非百分比）填充到共享内存区域，并注册字符设备 /dev/cpu_stat_monitor，用户空间程序可通过 mmap 将该共享内存映射到自己的地址空间

- 用户空间监控器（CpuStatMonitor）：

- 用户空间程序打开 /dev/cpu_stat_monitor 设备，通过 mmap 映射共享内存，实时读取内核模块采集的 CPU 累计时间数据
- 采集后与本地缓存中上一次的数据进行对比：

- 如果该 CPU 之前已有缓存数据，则计算两次采集之间的时间差值
- 分别统计总运行时间差（total_time = user + system + idle + nice + iowait + irq + softirq + steal）和忙碌时间差（busy_time = user + system + nice + irq + softirq + steal）
- 根据差值计算 CPU 整体使用率及各项细分使用率（user%、system%、nice%、idle%、iowait%、irq%、softirq%）

- 计算得到的百分比数据填充到统一的 protobuf 消息体（MonitorInfo）的 cpu_stat 字段
- 最后更新本地缓存（cpu_stat_map_），保留本次采集的原始累计值用于下一轮比较
- （首次调用 UpdateOnce() 时由于没有历史数据，不会产生 cpu_stat 输出，需要至少两次调用才能计算出有效的使用率百分比。）

- 数据在内核存储位置

- 在 Linux 内核中，每个 CPU 都有一个 struct kernel_cpustat 类型的 per-CPU 变量，其名称为 kstat_cpu。该结构体中包含一个 cpustat[] 数组，用于记录该 CPU 在不同状态下的累计时间（单位为纳秒）。内核提供了 kcpustat_cpu(cpu) 宏来访问指定 CPU 的统计数据。每个状态在内核中都有一个固定的枚举值（定义在 <linux/kernel_stat.h> 中），这些枚举值对应 cpustat[] 数组中的下标：

- `user`：用户态消耗的 CPU 时间，比如普通应用程序的运行时间。
- `system`：内核态消耗的 CPU 时间，比如系统调用、内核服务等。
- `idle`：CPU 空闲时间，没有任务需要处理时的时间。它不包括等待 I/O 的时间（iowait)。
- `nice`：以“低优先级”运行的用户进程消耗的 CPU 时间。也就是进程的 nice 值被调整为 1-19 之间时的 CPU 时间。nice 可取值范围是 -20 到 19，数值越大，优先级反而越低。CPU 调度器会减少其时间片分配。
- `io_wait`：CPU 等待 I/O 操作（如磁盘、网络）完成的时间。
- `irq`：CPU 处理硬件中断（如网卡、磁盘等设备中断）的时间。
- `soft_irq`：CPU 处理软中断（如网络包处理、定时器等异步任务）的时间。

- 内核数据统计逻辑

- 每当进程发生状态切换，比如从用户态切换到内核态，或者进入休眠、I/O 等状态时，内核调度器会根据进程当前的状态（如 TASK_RUNNING、TASK_INTERRUPTIBLE、TASK_UNINTERRUPTIBLE 等）。根据状态的不同会在合适的时机自动调用不同的统计函数，把本次消耗的时间累加到对应 `cpustat[]`项。比如，当进程处于用户态运行时，内核会在适当的时机调用 account_user_time()，把这段时间累加到当前 CPU 的 cpustat[CPUTIME_USER] 中；如果进程处于内核态，就会调用 account_system_time()，记录内核态的时间消耗。这些统计函数的调用点主要在时钟中断处理（如 `scheduler_tick()`）、进程上下文切换、进程状态切换和中断处理这几个内核的处理过程中。确保准确实时的统计。

- 以用户态时间统计为例，每次发生时钟中断时，内核会调用 scheduler_tick()。这个函数是调度器的定时心跳，用来维护进程调度和时间统计。在 scheduler_tick() 中，内核会判断当前进程是在用户态还是内核态。如果是用户态，就调用 account_user_time()，把这次时钟中断所占用的时间累加到当前 CPU 的 cpustat[CPUTIME_USER] 中。这样，每次时钟中断都会将当前进程所消耗的时间按状态分类，更新到对应的 cpustat[] 数组中，从而实现对 CPU 时间的分状态统计。
- 当进程因为 I/O 等原因进入休眠状态且 CPU 空闲时，内核会在 account_idle_time() 中根据是否有进程等待 I/O 来判断，将这段时间累加到 cpustat[CPUTIME_IOWAIT] 或 cpustat[CPUTIME_IDLE] 中。当 CPU 处于硬中断或软中断处理过程中时，内核也会调用相应的统计函数（如 irqtime_account_irq()），分别将时间累加到 cpustat[CPUTIME_IRQ] 和 cpustat[CPUTIME_SOFTIRQ] 中。所以，不管进程是运行在用户态、内核态，还是处于休眠、I/O 等待状态，内核都能准确地统计出它在各个状态下的时间消耗。
- 在支持 NO_HZ（tickless）模式的内核中，当 CPU 长时间空闲时会停止时钟中断，此时传统的基于 tick 的统计可能不够精确，需要通过 get_cpu_idle_time_us() 等接口获取更准确的空闲时间。

- 指标统计的意义

- 可以看到每个 CPU 的使用情况和负载来源，这样一旦出现性能问题，比如 CPU 占用高、系统卡顿。可以进一步去分析
- 通过分析 user、system等比例，区分是应用层压力大（比如业务进程消耗多），还是内核态消耗高（比如系统调用、内核服务占用多）
- io_wait、irq、soft_irq 这些指标可以帮助我们定位更具体的性能问题， 比如 io_wait 很高，说明 CPU 大量时间在等磁盘或网络 I/O，可能是存储或网络成为瓶颈；irq 或 soft_irq 很高，可能是中断特别多、网络压力大或者驱动异常导致的。

###### CPU为什么满了？如何排查？

通常情况下cpu满载的原因

- 首先，可能是业务进程本身计算量太大，比如做大量数据处理、算法运算、压缩解压或者加密，这些都会把 CPU 吃满。
- 也有可能内核态这边消耗高，比如系统调用特别频繁、有大量的内核模块的操作，或者是驱动异常导致内核态 CPU 使用率升高。
- 还有中断/软中断压力大。比如网络流量很大、磁盘 I/O很 密集，导致 CPU花了大量时间在处理中断或软中断（如网络包收发、定时器等）。
- 多线程程序中，如果锁的竞争严重，线程老是在争抢锁资源，cpu也会被白白浪费掉，实际有效利用率不高。
- 还有就是进程或者线程太多，远远超过 CPU 核心数，CPU 花了很多时间在切换任务上，调度开销就上来了。
- 还有一种情况是 cpu负载很高但cpu的利用率不高，负载高可能是因为磁盘或者网络太慢，很多进程都在等 I/O，这时候系统负载看起来很高，但 CPU 实际上是没怎么用的。

排查

通过综合看load、stat、softirq 这三类指标，排查 CPU 占用高的原因。

- 首先，看一下 CPU 的整体使用率，确认是不是所有核心都已经跑满了。
- 接着会看平均负载，比如 1 分钟负载如果大于 CPU 核心数，说明短时间内系统压力很大；如果 15 分钟负载也一直很高，说明系统长期处于高压状态。 有时候会遇到负载很高但 CPU 使用率其实不高，这种情况往往是 I/O 瓶颈，比如磁盘或者网络阻塞导致大量进程在等待。
- 然后进一步分析每个 CPU 的详细使用情况：

- 如果 `%us`（用户态）很高，说明是用户进程本身计算密集，可能是业务压力大或者有异常进程。
- 如果 `%sys`（内核态）很高，说明内核在忙，比如系统调用频繁或者内核模块有问题。
- 如果 `%soft`（软中断）很高，说明内核在处理大量软中断，常见于网络压力大或者中断风暴。
- 如果 `%iowait` 很高，说明 CPU 大量时间在等 I/O，通常是磁盘或网络瓶颈。

- 最后排查软中断的详细情况。

- `NET_RX/NET_TX` 如果很高，说明网络收发压力大，可能是网络瓶颈或者有异常流量。
- `SCHED` 如果高，说明调度压力大，可能是进程/线程太多或者调度器有问题。
- `RCU` 如果高，可能是锁竞争严重或者内核同步压力大。

通过这些去判断到底是业务压力、内核、I/O 还是中断导致的 CPU 满载，然后再去做相应的优化，比如优化代码、调整内核参数、升级硬件或者优化网络配置等。

补充

- 还可以去看查某台服务器CPU 相关指标的历史数据的变化趋势。比如发现某个时间点 CPU 占用突然升高，就可以结合业务变更、流量变化等信息，定位问题发生的具体时刻和背景。
- 然后看某个指标的变化率可以直观地发现异常，比如某个 CPU 的软中断占比在短时间内飙升很快，这往往是系统出现异常的早期信号。通过对比变化率，可以提前发现潜在的性能瓶颈或故障隐患，预防故障。

###### 内存信息的采集（/proc）

- 从/proc/meminfo中读取文件内容，文件为两列，前者为统计的属性，后者为对应的值，
- `proc/meminfo`存储的信息

- 重要的几个
- `total`系统总内存大小(KB)
- `free`当前未使用的空闲内存大小 (KB)
- `avail`可用内存，系统认为可以立即分配给新进程使用的最大内存，包含未使用的内存和可回收的缓存(KB)，所以一般比未使用内存更大。比 `MemFree` 更准确地反映“实际可用”的内存资源。
- `buffers`用于块设备缓冲区的内存大小。是内核用于缓存原始磁盘块的临时存储区域，是对磁盘数据的缓存，优化磁盘的写操作，可以把多次小的写合并成单次大的写，减少直接IO次数。
- `cached`用于文件缓存的内存大小。内核通过 Page Cache（页缓存） 缓存从磁盘读取的文件数据，是对文件数据的缓存，加速文件的读操作，这样子再次访问这些文件数据时，就可以直接从内存中快速获取，不需要访问磁盘。

- buffers和cached都属于可回收内存，当内存紧张时，内核应主动回收部分 缓存，提升 `MemAvailable`。

- `swapcached` 在swap交换分区中缓存的内存页数量。当系统内存紧张时，内核会将部分内存页换出到 磁盘的swap 分区/文件（`swap out`）。如果某个进程再次访问这些数据，内核会将其从 swap 加载回内存（`swap in`）。但内核并不会立即释放 这些在swap中的内存页的副本，而是标记为 `SwapCached`被缓存在swap交换区里，如果未来内存又不足了，就可以直接丢弃内存中的页面，不用像之前那样换出，因为swap已经有备份了，无需重复写入，可以减少磁盘写入的次数。本质是用磁盘空间换取更少的IO操作。

- `active`最近频繁使用的活跃内存 (KB)、`inactive`较少使用的非活跃内存 (KB)、
- `active_anon`活跃匿名页 ，一般在进程堆栈内存里(KB)、`inactive_anon`非活跃匿名页 (KB)，这部分优先缓存到swap交换分区，反映的是应用层的内存使用情况
- `active_file`、`inactive_file`、文件映射页面的活跃/非活跃状态

- `dirty`等待写入磁盘的脏页数 (KB)、`writeback`正在写回磁盘的内存页 (KB)
- `anonpages`匿名页面的内存使用总量 ，匿名页表示没有关联文件的内存页，通常用于堆栈、malloc 等动态分配。`mapped`映射页内存 (KB)、映射页表示与文件（如 mmap）关联的内存页。加载很多文件或使用共享库，`Mapped` 会增长。
- `kreclaimable`可回收的内核内存(KB)，包括 `SReclaimable`，也包括其他可回收的内核结构（如部分网络缓冲区等）、`sreclaimable`Slab 中可回收的部分，表示那些可以被回收的slab对象缓存，当内核内存压力大时就会被释放、`sunreclaim`Slab 中不可回收的部分

- Slab 是 Linux 内核中用于管理常用对象的的一种缓存机制，用于快速分配常用的数据结构

- 解析其中的关键字段，将这些字段对应的数据提取出来后，数值转换为 GB 单位，根据总内存和可用内存等信息计算内存使用率（total - avail) / total * 100）；把这些内存相关的信息写入统一的protobuf消息体的内存信息字段
- 选择读取 /proc/meminfo 的方式采集内存信息

- 内存相关的数据在内核中分布广，内核的方式复杂
- 内存的信息更新频率较低（随内存分配、释放、回收等事件实时更新），读取proc文件的方式可以满足实时采集的需求。内核模块更适合高频数据的采集
- /proc的文本数据不大，解析开销小可以接受
- meminfo 已经由内核统一整理和汇总了所有关键的内存统计项，不需要额外开发。通过读取这个接口，可以一次性、方便高效地获取到全面的内存使用情况。

###### 指标统计的意义 定位内存问题

- 首先，可以通过系统总内存，空闲内存，可用内存和缓存，内存使用率，查看系统整体的内存使用情况。看缓存是不是占用了过多的内存，看变化率有没有突变，帮助及时发现内存不足，提前预警可能存在的内存问题。
- 再通过查看一段时间的趋势判断出内存问题的类型，最后详细分析定位具体的问题所在。

- swap缓存可以分析内存交换的情况，当 `SwapCached` 值很高时，说明系统正在频繁地将内存页交换到磁盘，swap持续高还表示系统有内存压力，物理内存不足。一段时间内的升高可能会导致系统性能下降，因为swap 换出操作依赖磁盘 I/O，速度比内存访问要低很多

- 脏页（Dirty）、写回页（Writeback）来分析 磁盘I/O 压力，如果 `Dirty` `writeback`持续高值，表示大量内存页等待写入磁盘，磁盘处理不过来 ，会存在 磁盘I/O 性能瓶颈，脏页积压，导致系统卡顿

- 匿名页面`anonpages`可以看应用层的内存消耗，主要是应用程序堆栈、malloc等动态分配的内存。可以再看映射页分析应用层是否有内存泄漏，如果匿名页面有异常增长而映射页不变，（排除文件映射的影响）说明程序在不断地申请匿名内存但是没有释放，可能有内存泄漏。
- slab内存 可以看内核内存使用，是内核的对象缓存机制，分为可回收（SReclaimable）和不可回收（SUnreclaim）。若 内核中不可回收的部分持续增高且无法释放，说明内核层面可能存在内存压力。

###### 网络信息的采集 （ebpf）

- 挂载点选择

- XDP（eXpress Data Path）：挂载在网卡驱动的最早入口，适合极低延迟和只需统计入方向的场景。
- TC（Traffic Control）：挂载在协议栈的 ingress/egress，适合需要更丰富上下文和双向统计。

- 实际开发中，一般选择 XDP 层挂载，能最快拿到每个数据包，统计最精确。（但是XDP 只能统计入包，TC 可以统计出包）。这里挂载的 hook 点是每个物理网卡的 TC（Traffic Control）层的 ingress 和 egress，也就是协议栈进入和离开网卡的两个方向。这样可以分别统计入方向和出方向的流量，适合需要统计收发流量的场景。

- 编写 eBPF 程序

- eBPF 程序被加载到内核后，会在网络包到达时触发执行；
- 每收到一个数据包，eBPF 程序通过 skb->ifindex 获取网卡索引（ifindex），用户空间程序通过 if_indextoname() 将 ifindex 转换为网卡名称
- 每当一个网络数据包被内核处理时，eBPF 程序会根据数据包的入方向（ingress）或出方向（egress）来分别统计相应的指标。
- 对于入方向（ingress），统计接收字节数 (rcv_bytes)、接收包数 (rcv_packets)；对于出方向（egress），统计发送字节数 (snd_bytes)、发送包数 (snd_packets)
- 所有统计数据都存储在 eBPF 的哈希表 net_stats_map 中。该 map 的 key 是网卡索引 ifindex（__u32 整数类型），value 是我们定义的结构体 struct net_stats，包含四个字段：rcv_bytes、rcv_packets、snd_bytes、snd_packets，用于记录每个网卡的流量状态。
- eBPF map 是内核与用户空间共享的哈希表，可以快速查找和更新，
- 表类型的选择（hashmap）

- 用 BPF_HASH 作为 eBPF map 类型，是因为每个网卡都需要独立的存储，网卡名是动态的 、不连续的key，哈希表能支持按 key 查找和更新。更适合这种“每个对象一份状态”的统计场景。
- 数组（BPF_ARRAY） 只能用数组下标访问，不适合用 ifindex 这种不连续的整数作为 key（ifindex 可能是 1, 2, 5, 17 等不连续值，会浪费大量空间）。适合固定数量、下标已知的对象，不适合动态数量的设备统计
- 队列/栈（BPF_QUEUE/BPF_STACK）/环形缓冲区（BPF_RINGBUF）  
    实现了先进先出（FIFO）和后进先出（LIFO）的数据结构，适合在内核和用户空间之间传递事件流或日志记录，比如把一批事件按顺序推送到队列里，用户空间再按顺序取出来处理，不适合做按 key 的状态统计和查找。
- LRU 哈希表（BPF_LRU_HASH）  
    BPF_LRU_HASH 会在表项数量超过上限时，自动淘汰最久未被使用的 key，以保证内存不会无限增长。这种结构适合 key 数量非常大、且 key 经常变化的场景，正常机器网卡数量有限，普通哈希表已能满足网卡统计需求，更加稳定。

- eBPF 程序加载与挂载

- 用户空间通过 libbpf C API 和 BPF skeleton 机制加载写好的 eBPF 程序，并将其挂载到各个网络接口上，实现流量采集功能。

- libbpf 是 eBPF 生态里轻量级的 C 库，用于加载和管理 eBPF 程序。配合 bpftool 生成的 skeleton 头文件，可以很方便地在用户空间加载 eBPF 程序、管理 eBPF map，并与内核态的数据进行交互。

- 用户空间遍历 /sys/class/net/ 目录，过滤掉 lo、虚拟网卡等，仅保留所有物理网卡
- 为每个物理网卡创建对应的 eBPF 程序实例；
- 将 eBPF 程序分别挂载到每个网卡的 ingress 和 egress hook 点上，确保所有流量都能被捕获并统计
- eBPF 程序通过 skb->ifindex 获取网卡索引，作为 key 存入 net_stats_map 中；用户空间通过 if_indextoname() 将 ifindex 转换为网卡名
- 用户空间通过 libbpf 提供的 bpf_map__fd() 接口获取 map 的文件描述符，再通过 bpf_map_lookup_elem() 和 bpf_map_get_next_key() 遍历和读取内核 eBPF map 中的统计数据

- 用户空间采集数据

- 用户空间定时（10秒）通过这个 map 句柄批量读取所有网卡的统计数据，包括网卡名、接收和发送的字节数、包数

- 速率计算

- 读取到原始数据后，我们会结合上一次采集的结果和时间点，计算每个网卡的流量速率和包速率。
- 最后把这些数据填充到 protobuf 消息体中的netinfo字段。

###### 选择eBPF的原因

- 调研

- 像Facebook、Google、RedHat 这些大厂都在用 eBPF重构他们的基础设施，说明这一技术成熟而且是有性能优势和安全优势的，尤其是在网络和性能监控方面。
- Facebook：他们用 eBPF 重写了他们的网络基础设施，比如用 eBPF 替代了传统的 iptables 和 netfilter，提升了网络包过滤和转发的性能。Facebook 还将负载均衡从 IPVS 迁移到基于 eBPF 的方案，实现了更高的可编程性和可维护性。
- Google：Google 利用 eBPF 进行系统 Profiling，能够在分布式系统中精确分析应用的 CPU 消耗。此外，Google 也在探索用 eBPF 来进行流量分析和网络安全能力
- RedHat：RedHat 推出了 bpfilter 项目，用 eBPF 替换原有的iptables实现，进一步提升了包过滤的效率和可扩展性
- 与其他方式的对比

- /proc 方式，存在一定的延迟
- 网络流量是高频事件，每秒可能有成千上万的数据包到达，而且每个数据包都可能来自不同的网卡、不同的方向（入方向和出方向），我们需要对这些数据进行实时的统计。内核模块mmap和/proc方式
- mmap 虽然也较高效，但通常用于统计 CPU、软中断等静态数据，不适合网络流量数据的采集。而且因为是内核模块，安全性低

###### 网络指标统计的意义

首先看网络流量速率变化率，出现突增或突降，就是有异常情况了，可能有异常流量或网络攻击，可以结合历史数据看是否超过正常波动范围。然后还可以结合软中断，如果网络收发软中断频率持续升高，但网卡流量不高，说明协议栈处理压力大，可能是小包太多（总流量不大但每秒要处理的包数量很多）。还有，如果接收错误数持续出现不为零，可能是着网卡驱动有问题；而收发丢包数 增长，则表示内核网络栈处理的问题或者应用层处理不过来。最后还可以通过检查各个网卡的负载是否均衡，如果某个网卡流量远高于其他网卡，系统配置有问题

###### 磁盘IO信息的采集

- 读取 /proc/diskstats 文件，每一行对应一个磁盘设备。解析出关键字段，字段主要包含读写次数、扇区数、io耗时等，通过采样间隔计算得到读写速率、平均延迟、利用率信息，采集后填充到 protobuf 的 DiskInfo 字段。
- 原始存储字段数据（第三个字段开始）

- `name`（磁盘名）
- `reads/writes`（成功完成的读写操作次数）
- `sectors_read/sectors_written`（累计读取写入到的扇区数）

- 每个扇区512字节，sectors_read * 512 = 读取的总字节数（单位 byte）

- `read_time_ms/write_time_ms`（读写操作累计耗时，毫秒）
- `io_in_progress`（当前进行中的IO操作数）
- `io_time_ms`（磁盘累计处于IO操作的时间，毫秒）
- `weighted_io_time_ms`（加权IO操作时间，毫秒）

- 填充字段
- `read_bytes_per_sec` / `write_bytes_per_sec`：每秒读写吞吐量（KB/s）。字节数差值/时间间隔/1024
- `read_iops` / `write_iops`：每秒完成的 I/O 次数（次/s）.成功读写次数差值/时间间隔
- `avg_read_latency_ms` / `avg_write_latency_ms`：平均 I/O 延迟（毫秒）。读写累计耗时/读写操作数
- `busy_time_percent`：磁盘繁忙时间占比（%）。累计io时间/时间间隔 ×100%

###### 磁盘指标统计的意义

- 先看使用率

- 一般超过 60%，说明磁盘压力大；
- 如果低于 30% 但延迟高，问题可能不在磁盘本身，可能是文件系统或应用程序的问题（比如过多的同步写操作）

- 再看延迟

- HDD 机械硬盘正常应在几毫秒到十几毫秒之间；
- SSD固态硬盘 一般应低于 2ms；
- 如果超过就需要注意，如果使用率很高，延迟也高一般来说是是正常的（磁盘忙）；但如果使用率低而延迟高，→ 可能是同步写操作或文件系统配置问题。

- 结合 IOPS 和吞吐量判断 I/O 模式

- 高IOPS + 低吞吐量：说明每次I/O操作的数据量很小（比如小于4KB），这通常是随机小I/O，常见于数据库、日志等场景。这种场景对磁盘的随机性能要求高，如果是HDD机械硬盘的话可能是瓶颈。
- 低IOPS + 高吞吐量：说明每次I/O操作的数据量很大（比如超过64KB），顺序IO多，如大文件读写，磁盘利用充分。
- 两者都高：通常是高负载场景，需要关注磁盘是否能够承受。

##### 主机信息标识采集

- 主机名获取：使用 POSIX 标准的 `gethostname()` 系统调用获取主机名；

- IP地址获取：使用 `getifaddrs()` 遍历所有网络接口，过滤规则：

- 只获取 IPv4 地址（`AF_INET`）
- 跳过 loopback 接口（`lo`）
- 跳过虚拟网卡（`docker*`、`veth*`、`br-*`、`virbr*`）
- 返回第一个有效的物理网卡 IP

##### 相关知识点

###### proc文件系统

- proc 是 Linux 位于内存中的虚拟文件系统（procfs），所有内容存储在内核内存，但不占用磁盘空间，存储的是内核和进程的各种实时信息，大致包含了内存管理、每个进程的相关信息、文件系统、设备驱动程序、终端、系统控制参数、网络等信息。

- 它为内核空间和用户空间之间打开了一个通信窗口。`/proc`目录下的大部分文件都是只读的，部分文件是可写的，用户可以通过这些可写的文件来修改内核的一些配置;。也可以通过这些文件获取系统及当前正在运行进程的实时信息。而且用户每次读取时都会重新实时地生成内容，不会持久化，也不会缓存，是内核根据当前运行状态动态生成的

- 相当于以文件形式提供了一个指向内核数据结构的接口，`/proc` 下的每个文件都关联了内核回调函数（如 `file_operations.read`），当用户空间进程通过 open/read 等系统调用访问 proc 下的文件（如 stat、meminfo、loadavg），内核通过VFS（虚拟文件系统）层调用这些函数，实时从内核数据结构获取最新数据，格式化为文本，并通过 `copy_to_user` 拷贝到用户空间缓冲区，用户进程从buf中读取数据。例如，`top`、`htop`、`free`、`ps` 等命令，都是通过读取 proc 下的相关文件获取系统和进程的实时状态，进一步换算得出的。
- 用户空间和内核空间的数据交互方便，且由于是虚拟文件系统，读取其中的内容不会影响磁盘 I/O 性能，也不会显著增加系统负载，更不会产生持久化数据，安全高效。
- **存储的特殊子文件或子文件夹**

- 进程目录，以数字命名，当前运行的每一个进程都有对应的一个目录在/proc下，以进程的 PID号为目录名，它们是读取进程信息的接口。`/proc/[pid]/` 目录下是每个进程的详细信息
- /cpuinfo CPU 型号、核心数、频率等
- /devices 已经加载的设备并分类
- /filesystems 内核当前支持的文件系统类型
- /interrupts x86架构中的每个IRQ中断数
- /iomem 系统中所有物理设备的当前内存映射情况
- /locks 内核锁住的文件列表
- /modules 所有加载到内核的模块列表
- /mounts 系统中使用的所有挂载
- /partitions 磁盘分区中的块分配信息
- /version Linux内核版本和gcc编译版本
- /driver 驱动信息
- /fs 文件系统信息
- /vmstat 虚拟内存统计信息
- /diskstats 取得磁盘信息
- /schedstat kernel调度器的统计信息
- /stat 提供了 CPU使用情况统计信息，/meminfo 提供内存使用情况。/loadavg提供负载状态信息。/net/dev存储网卡设备流量信息。/softriq提供软中断信息。

###### 内核模块

- Linux内核模块是用于动态扩展内核功能的一些代码。本质上是`.ko`格式的文件，内核模块通过与内核链接，实现对内核功能的扩展。模块动态加载生成一个后缀为`.ko`（Kernel Object）的内核模块文件。我们需要使用此内核模块时，可以使用加载内核模块命令（`insmod`/`modprobe`）把内核模块加载到Linux内核中，不需要使用此模块时，可以使用卸载内核模块命令（`rmmod`）把内核模块从Linux内核中卸载，加载操作和卸载操作都无需重启内核。编写的内核模块构建并动态加载到内核中，运行在内核空间，它可以访问内核的全局变量、函数、数据结构等，就像内核的一部分一样。还可导出符号供其他内核模块使用
- 内核模块编写

- Linux内核开发所必需的头文件（3个，init、module、kernel），提供模块定义、初始化退出的功能
- 声明模块的许可证，`module_license`，

- 可以使用MODULE_AUTHOR() 和 MODULE_DESCRIPTION() 来添加作者信息和模块描述。

- 编写init（加载）函数和exit（卸载）函数。

- init 函数用于初始化资源，比如注册字符设备、创建定时器、分配内存等；exit 函数用于释放资源，比如注销设备、销毁定时器、释放内存等；

- 调用 module_init 和 module_exit 函数告诉内核哪些函数是内核模块的初始化和退出函数，

- 加载和卸载模块的时候内核自动调用

- 实现逻辑，注册字符设备、定时器、数据采集，要与用户交互还要实现`file_operations` 结构体相关回调（如 `.read`、`.write`、`.mmap` 等）
- 编写makefile，make编译模块
- 使用insmod将编译的模块加载进内核进行测试

###### 字符设备

- 在 Linux 内核中，字符设备是一种内核与用户空间进行传递消息的接口机制。它通过设备文件（如 /dev/xxx）暴露给用户空间，允许用户程序通过系统调用（如 open()、read()、write()、mmap() 等）与内核模块通信。字符设备的核心是 file_operations 结构体，相当于一个接口，它定义了一组回调函数指针，比如 .read、.write、.mmap 等，可以在这里实现这些回调函数，定义行为。
- 在我们的项目中，在内核模块中分配了一块内存，用于存储 先前获取到的统计数据在，然后创建一个字符设备节点，并实现 `file_operations` 结构体中的 `.mmap` 回调。这样用户空间通过open打开整个设备文件，获得一个文件描述符，进而实现将内核模块的这块内存映射到自己的虚拟地址空间，实现对内核数据的访问。

本质上是一个文件，通过文件I/O就可以对设备进行操作，字符设备的数据逐字节传递，

###### mmap内存映射

- 通常用户空间的 mmap 只能针对“文件描述符”，而普通文件只能映射磁盘文件内容，无法直接映射内核内存。这里内核模块注册的字符设备，可以自定义`mmap`回调，把内核内存区（如结构体、缓冲区）映射到用户空间，实现内核态与用户态数据共享。
- mmap内存映射的方式直接与内核进行交互获取数据，用户空间可以通过`mmap`直接映射内核分配的物理内存，用户态和内核共享一块物理内存，数据更新后用户空间可直接读取这块内存，不需要每次都走系统调用和拷贝。实现“零拷贝”，高效、低延迟，非常适合实时监控、数据采集等场景。
- 原理

- 用户空间调用 mmap，内核在进程的用户虚拟地址空间分配一段虚拟内存区域如 。mmap（）还会返回一个指针，指向进程虚拟地址空间的一个地址。这时，虚拟地址和物理页的映射关系还没有真正建立（即还没有分配物理页，也没有建立页表项），只是做了虚拟内存的预留。当用户空间第一次访问映射区域的某个地址的时候，CPU 发现该虚拟地址没有对应的物理页（页表项无效），于是触发缺页异常（page fault）。操作系统内核捕获到这个异常，进入内核态，查找该虚拟地址属于哪个映射区，然后建立虚拟地址到物理内存的真实映射，建立页表项，恢复用户态的执行，用户空间后续访问该虚拟地址，就会直接命中页表，访问的就是内核的映射区物理内存。

## 监控信息管理

这里是管理者服务器的功能

### 与多台工作服务器交互获取数据

每台工作服务器会创建一个gRPC客户端（MonitorPusher），每隔十秒主动采集本机监控数据，并通过 SetMonitorInfo 接口推送给管理者服务器。管理者服务器运行 gRPC Server（GrpcServerImpl），接收来自各工作服务器的推送请求，并将数据存储到内存中

### 数据处理并存储

管理者服务器对每台工作服务器推送的数据进行性能评分，计算变化率，然后将每台服务器的监控数据及评分、时间戳等信息写入 MySQL 数据库，所有历史数据都会存储。若某台工作服务器推送失败，工作服务器端记录日志并在下一周期重试，不影响其他服务器的数据接收

##### 性能变化率

- 计算逻辑

- 每次写入数据库之前获取当前拉取到的各项指标，然后和上一次相同服务器采集到的做对比，得到变化率。相对变化率 = (当前值 - 上一次值) / 上一次值100%。

- 变化率指的是某个性能指标在相邻两次采样之间的相对变化速度。
- 变化率反映指标的趋势和波动，比原本的绝对值更能看出系统性能的的突发异常和趋势性问题，例如，CPU使用率突然大幅上升，变化率会很大。网络接收速率变化率突然增长很多，流量突发增长。内存增长速率猛增，要去检测是不是存在内存泄漏
- 通过查询变化率，

- 用户可以看到某台服务器在某段时间内的性能变化趋势（CPU利用率、内存占用、网络流量），比如是否有突发的资源消耗、是否存在持续的性能下降等，很清晰地发现性能瓶颈，负载高峰等现象，提前预警可能的故障发生
- 当出现故障或性能下降时，可以回溯历史变化率，定位问题发生的具体时间点和相关指标的异常变化。结合多项指标的变化率，可以辅助判断是 CPU、内存、磁盘还是网络导致的性能问题。

#### 系统性能评分实现

拿到每个工作服务器的性能数据之后，选取几个指标设计一定的权重对实时系统性能进行评估，根据业务场景设定不同的权重指标，然后综合计算出性能得分。

##### 性能评分模型设计

###### **基础性能指标**

项目使用CPU整体使用率、最近一分钟的CPU负载、内存使用率、网络收发字节数的速率作为基础指标，可以根据实际场景需要加入其他的指标，比如软中断频率

###### **得分计算**

各个基础性能的权重根据业务场景灵活配置，这里学校网站的实际负载高的时候主要是选课系统开放的时间、查成绩的时间，登录查询的请求量会达到平常的十倍以上。cpu使用率设置为35%（高并发计算量大，使用率越高，响应延迟越明显），内存使用率设置为30%（多用户同时在线访问需要缓存课程/成绩数据，过高易触发 swap硬盘交换数据，显著降低性能），cpu负载15%，磁盘io15%，网络带宽5%（以数据库交互为主，带宽压力小），然后将每个权重归一化之后进行加权得到最终的分数，归一化处理使用反向归一化，这样子使用率越低的得分会越高，反之分数越低就表示系统资源越紧张。

###### **具体实现**

**反向归一化**

|   |   |   |
|---|---|---|
|指标|符号|反向归一化公式|
|CPU使用率|_P_CPU|_S_CPU=1−_P_CPU/100|
|CPU负载|_P_LOAD|_S_LOAD=1−_P_LOAD/_P_LOAD,max|
|内存使用率|_P_MEM|_S_MEM=1−_P_MEM/100|
|网络收速率|_P__INET_|_S__I_NET=1−_P__INET_/_P__INET_，max|
|网络发速率|_P__ONTE_|_S__O_NET=1−_P__ONTE_/_P__ONTE_，max|
|磁盘IO利用率|_P__DISK_|_S__DISK__=（1-P__DISK__）*100%_|

CPU的最大负载通常取CPU物理核心数 × N（`N` 是经验系数，一般取 `1.0~2.0`），根据实际的业务场景调整，如CPU密集型的计算任务，取`N=1.0`；I/O密集型（如数据库、网络服务），可放宽至 `N=1.5~2.0`（因负载可能包含等待I/O的进程）。**最大带宽**通常取 历史峰值流量 × 缓冲系数。理论最大值：由网卡规格决定（如1Gbps网卡的理论上限为125MB/s）

**综合得分公式**

**Score=0.35×S****CPU****+0.30×S****MEM****+0.15×S****LOAD****+0.15×S****DISK****+0.05×(S****INET****+S****ONET****)/2**

  

**展开形式：**

**Score=0.35(1−**_P_CPU**/100)+0.30(1−**_P_MEM**/100)+0.15(1−**_P_LOAD**/**_P_LOAD**,max****)+0.15(1−**_P__DISK_**/100)+0.025(1−**_P__INET_**/**_P__INET_**,max****)+0.025(1−**_P__ONTE_**/**_P__ONTE_**,max****)**

### 功能性验证

验证性能评分逻辑计算到的值是准确的，分数能够真实准确地反映出服务器的实际运行状态，通过实际负载测试，确保分数跟服务器的真实状态即是否卡顿是对应的

##### 实际验证方法

- 本机验证，两台虚拟机
- 使用 `stress`、`iperf` 等工具模拟和验证不同负载场景。

- 通过 stress 工具模拟高 CPU 负载；stress --cpu 4 --timeout 60s，跑满四核cpu
- 模拟内存高负载：stress --vm 1 --vm-bytes 2G --timeout 60s
- 模拟磁盘 I/O 压力；stress --io 4 --timeout 60s
- 使用 stress 模拟指定的磁盘 I/O 压力；stress --io 4 --timeout 60s
- 通过 iperf 模拟高网络带宽压力；启动服务端（在一台机器上运行）：iperf -s；客户端发起 TCP 流量测试（在另一台机器上运行）：iperf -c <server_ip> -b 10M -t 60 -i 1，-b 10M：设定带宽为 10MB/s；

- 查询数据库，观察分析各时刻分数变化与实际服务器体验的对应关系。

##### 实际场景

- 空闲状态：服务器无业务或轻负载，CPU/内存/网络占用低，负载低。此时分数应接近 100。
- 正常状态：服务器有正常业务，资源占用适中。分数应在 70-90 区间。
- 轻度卡顿：CPU、内存、网络等资源占用较高，负载接近或略高于最大值。分数应在 50-70 区间。
- 严重卡顿：资源接近满载，负载远超最大值。分数应低于 50。

- 在高负载期间，模拟用户请求最优服务器，观察返回结果。验证返回的服务器不是当前高负载节点，且分配逻辑按照评分逻辑

#### docker

##### 使用难点

- 搭建分布式环境与源码依赖的构建  
    难点：分布式项目需要多个服务协同运行，环境一致性要求高。很多依赖（如gRPC、protobuf、abseil等）需要源码编译，不能直接apt/yum安装。

- 先去看gRPC的的官方文档，有cmake 构建源码的文档，看怎么去进行源码安装构建的，理清所有源码构建需要的依赖和构建顺序，一步一步去构建。
- 遇到依赖缺失、编译报错时，去Google/GitHub查找解决方案，补齐依赖就可以。

- Dockerfile的编写，如何一步一步去调试  
    难点：Dockerfile需要一步步调试完善，调试时常遇到依赖缺失、构建顺序错误等问题。

- 构建时遇到缺少一些依赖的库文件的报错，就在Dockerfile中去添加安装命令，逐步补全，dockerfile是一步一步去添加完善的 。
- Docker有缓存机制，避免缓存失效：源码构建是有顺序的， 要在构建失败的那里，添加缺少的东西， 因为前面已经构建的是会有缓存的，再次编译构建就会直接从上次失败的地方开始 如果添加在本来构建失败的前面就会导致前面已经缓存好的失效，导致再次dockerbuild构建的时间变得很长，所以不能随意修改dockerfile的文件
- 小困难

- 优化镜像大小：源码构建完之后发现镜像文件占了十几个g的大小 占的硬盘大小挺大的 不合理 项目不大占这么多 就思考哪里占了空间 要把不需要的东西删除掉 减少镜像文件的大小 所以源码编译后之后rm-rf 删除了源码压缩包 和源码解压缩的一些临时文件，只保留最终产物
- Dockerfile 是会把本地的安装脚目录通过 `COPY` 指令拷贝到镜像的某个临时目录（比如 `/temp/install`），所以在后面 `RUN Install`执行安装脚本时，要先 `cd` 到镜像内的对应目录，否则会因为路径不对导致脚本找不到或者依赖文件缺失，安装失败。
- 构建镜像时，安装grpc之前要先安装好protobuf和abseil 当时只想着时用到了grpc和proto 只记得先装proto了，编译时报错发现还要装一个abesil库，后来查文档和社区才知道 abseil 是 gRPC 的底层依赖，于是又去 GitHub 上找 abseil 的源码，补充了 abseil 的安装步骤
- 构建依赖的时候使用的ubuntu自带的国外源会卡顿，就要修改成国内的镜像源来进行操作

- docker run docker into 脚本的编写

- 容器重命名会引起冲突：docker run里面会--name指定容器名字，这个名字只能用在一个容器上面，没删除酒容器又同名启动就相当于两个容器用了一个名字，就会报错，脚本无法继续执行
- docker run运行容器的脚本里面要先添加`docker stop`在执行`docker rm -f`停止和删除上一个容器，没有的话就要报错只用 `docker stop` 停止容器还不够，容器对象依然存在，名字被占用。必须用 `docker rm` 删除容器，确保每次启动前都同名容器被清理掉了
